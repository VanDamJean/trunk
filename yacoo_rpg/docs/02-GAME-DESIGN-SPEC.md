## 1. 전투 (~~하이브리드 모델~~ → 폐기됨)

> ⚠️ **이 섹션(§1 전체)은 폐기되었다.** 전투 모델은 **수동 턴제**로 변경되었으며, 최종 사양은 **`06-COMBAT-MANUAL-TURN-SPEC.md`** 를 따른다.
> 아래 하이브리드 설명은 히스토리 보존용으로만 남겨둔다. 구현 시 참고하지 말 것.
>
> 단, **§1.4 데미지 공식**과 **§1.5 가변 주사위 개수 족보 판정**은 전투 모델과 무관하게 여전히 유효하다(수동 턴제에서도 동일하게 사용).

### 1.1 기본 개념 (폐기)
전투는 **자동으로 흐르는 오토배틀(평타)** 위에 **주사위 필살기(궁극기)** 를 얹는 구조다. 기존 `CombatScreen.tsx`의 골격을 재사용한다.

- **오토배틀(평타)**: 영웅과 적이 각자 일정 주기로 자동 공격을 주고받는다. 플레이어 조작 없이 진행. (현재 구현 유지: 영웅 `heroAttackMs`, 적 `enemyAttackMs`.)
- **필살기 게이지**: 시간 경과(그리고/또는 평타·피격)로 채워진다. 가득 차면 "Open Dice" 활성화. (현재 `diceChargeMs` 기반 → 게이지화로 발전.)
- **주사위 필살기**: 게이지가 찼을 때 플레이어가 주사위를 굴려(최대 `maxRolls`회, 홀드 가능) 족보를 만들고, 한 족보를 선택해 **강공격**을 가한다. 사용 후 게이지 리셋.

### 1.2 전투 1회(노드 전투) 흐름
```
전투 시작 (영웅 HP = run.hp 로 시작, 적 HP = 노드/챕터 기반)
  while (영웅 HP > 0 && 적 HP > 0):
    - 평타 틱: 영웅→적, 적→영웅 자동 데미지
    - 필살기 게이지 충전
    - 게이지 가득 → 플레이어가 Open Dice
        · 주사위 굴리기(≤ maxRolls), 원하는 주사위 홀드
        · 유효한 족보 중 하나 선택 → 필살기 데미지 적용 → 게이지 리셋
  적 HP 0 → 승리: run.hp 갱신(현재 영웅 HP 유지) → 보상 단계로
  영웅 HP 0 → 패배: 런 종료(사망)
```

### 1.3 주사위 규칙
- 주사위 개수 = `run.diceCount` (런 중 가변, 2~5). **현재 `COMBAT_TIMING.diceCount=5` 고정 → 런 상태에서 받도록 변경.**
- 리롤 횟수 = `run.maxRolls` (런 중 가변). 첫 굴림 포함 카운트 방식은 현재 구현(`rolls < maxRolls`) 유지.
- 홀드: 굴린 주사위를 클릭해 고정, 고정 안 한 것만 다시 굴림 (현재 구현 유지).
- 족보 판정: `yahtzee.ts`의 `getValidAttackCategories`/`evaluateHand` 재사용. 단, **주사위 개수가 5 미만일 때의 족보 판정 규칙을 정의/확장해야 함**(아래 1.5).

### 1.4 데미지 공식 (★ 핵심 변경)
현재는 눈값을 무시한다(`calculateSelectedCategoryDamage` = `attack × multiplier × (1+diceBonus)`). 딸깍 다이스 손맛을 위해 **주사위 눈값 합**을 반드시 반영한다.

**필살기 데미지(제안 공식):**
```
ultimate =
  floor(
    (pipSum + weaponAttack)
    × handMultiplier
    × (1 + diceBonus + traitDamageBonus)
    × specialtyMultiplier
    × treasureHandMultiplier
  )
  + treasureFlatBonus
```
- `pipSum` = 선택한 족보에 기여하는(또는 전체) 주사위 눈의 합. **단순화를 위해 전체 5개 눈 합으로 시작해도 됨**(튜닝 대상).
- `weaponAttack` = `HERO_BASE.attack + 무기 레벨 기여분` (기존 `getHeroStats().attack`).
- `handMultiplier` = `HANDS[hand].multiplier` (기존 값 재사용).
- `diceBonus` = 부적(charm) 기여 (기존 `getHeroStats().diceBonus`).
- `traitDamageBonus` = 보유 특성들의 데미지 증가 합 (M3).
- `specialtyMultiplier` = 선택 족보가 무기 특화 족보(`weapon.specialtyHand`)와 일치하면 `>1`, 아니면 `1` (M3).
- `treasureHandMultiplier` / `treasureFlatBonus` = 보물 효과 (M3). M1~M2 단계에서는 `1` / `0`.

**평타 데미지(오토배틀):** 현재 `calculateHeroAutoDamage`(= attack + autoBonus) 유지. 적 평타는 `max(1, enemy.attack - defense)` 유지.

> M1에서는 특성/보물/특화가 아직 없으므로 그 항들을 중립값(1/0)으로 두고 **눈값 + 족보 배율**만 먼저 반영한다. M3에서 나머지 항을 채운다.

### 1.5 주사위 개수 < 5 일 때 족보 판정
주사위 개수가 런 중 2~5로 변하므로, `yahtzee.ts`의 판정이 가변 길이를 지원해야 한다.
- 현재 함수들은 `dice.length !== 5`면 무효 처리한다. → **길이 의존을 `run.diceCount`(또는 인자) 기준으로 일반화**한다.
- 가변 길이 족보 규칙(제안):
  - pair/twoPair/threeKind/fourKind/yahtzee: "n개 같은 눈" 기준은 동일하되, yahtzee=`diceCount`개 전부 동일, fourKind=4개 이상 등 가능한 경우만 활성.
  - smallStraight/largeStraight: 주사위가 4개 미만이면 비활성(혹은 길이에 맞춰 정의). 구현자가 규칙을 명확히 코드+테스트로 고정할 것.
- **반드시 단위 테스트로 각 주사위 개수별 판정을 고정**할 것.

## 2. 성장 구조 (런 + 메타)

### 2.1 런(run) — 로그라이크
- **런 시작**: Home에서 진입. `RunState` 새로 생성(시작 HP, 시작 주사위 수/리롤, 빈 특성/보물).
- **진행 단위 = 노드**: 챕터 맵의 노드를 하나씩 클리어하며 전진.
- **HP 유지**: 전투 사이에 영웅 HP는 유지된다(회복 노드/보상으로만 회복).
- **수집물은 런 한정**: 특성·보물은 이번 런에만 유효, 사망 시 폐기.
- **사망**: 전투에서 HP 0 → 런 종료 → 메타 정산.
- **챕터 클리어**: 보스 노드 처치 → 다음(더 어려운) 챕터로. "런+메타 둘 다" 방침에 따라 챕터는 사실상 무한 진행이며, **최고 도달 챕터를 메타에 기록**한다.

### 2.2 메타(meta) — 영구
- **코인**: 영구 재화. 장비 강화에 사용(기존 Upgrade 화면 유지).
- **장비 강화**: 무기/방어구/부적/신발 영구 레벨업(기존 시스템 유지).
- **해금**: 특성/무기 등을 영구 해금 → 이후 런의 보상 풀에 등장.
- **최고 기록**: 최고 도달 챕터 등.
- **메타 정산**: 런 종료 시, 런 성과(도달 노드/챕터, 처치 수 등)를 코인/해금 진행으로 환산해 `MetaSave`에 반영.

### 2.3 챕터 / 노드
- **챕터** = 노드 시퀀스(또는 간단한 분기 맵), 마지막은 보스 노드.
- **노드 타입(제안)**:
  - `battle` — 일반 전투
  - `elite` — 강한 전투, 더 좋은 보상
  - `treasure` — 전투 없이 보물/특성 획득
  - `rest` — 회복(HP 회복)
  - `boss` — 챕터 종료 보스
  - (선택, 후순위) `shop` — 런 내 임시 재화로 구매
- **노드 맵 생성**은 `chapter.ts`(신규)에서 RNG로 생성하되 **RNG 주입**(재현 가능)할 것.
- 적 스케일링: 현재 `createEnemy(stage)`를 `createEnemy(chapter, nodeType, rng)` 형태로 확장. HP/공격이 챕터와 노드 타입에 따라 증가.

## 3. 보상 3택1

전투(또는 보물 노드) 승리 후 **3개 후보 중 1개 선택** 화면(`RewardPickScreen`).
- 후보 풀: 특성(trait), 보물(treasure), 즉시효과(회복/임시재화 scrap 등).
- 후보 생성은 RNG 주입. 등급(희귀도)에 따른 가중치 적용.
- 선택 결과를 `RunState`에 반영(특성/보물 추가 또는 즉시효과 적용).

## 4. 특성(Trait) — 패시브 카드 (M3)

- **등급**: common / rare / epic / legendary (등급이 높을수록 등장 확률↓, 효과↑).
- **효과 예시**(스택 가능):
  - 시작 주사위 수 +1 (런 시작 시 `diceCount` 증가)
  - 리롤 횟수 +1
  - 특정 숫자(예: 6)의 눈값 가중치↑ 또는 특정 숫자 데미지↑
  - 특정 족보 배율↑
  - 전체 데미지 +x% (`traitDamageBonus`)
  - 평타 강화 / 받는 피해 감소 등
- **데이터**: `traits.ts`(신규)에 풀 정의 + `applyTraits(stats, traits)` 순수 함수.
- **해금**: 일부 특성은 메타 해금 후에만 런 풀에 등장.
- **(선택) 강화**: 같은 특성 중복 획득 시 레벨↑ 또는 효과 누적.

## 5. 보물(Treasure) (M3)

- **효과**: 각 조합(족보) 공격력 배율↑, 특정 숫자 공격력↑ (`treasureHandMultiplier`, `treasureFlatBonus` 또는 per-face 보너스).
- **데이터**: `treasures.ts`(신규)에 풀 정의 + 데미지 공식에 항으로 합류.
- **(선택) 강화**: 등급/레벨로 효과 증가.

## 6. 무기 특화 (M3)

- `EquipmentItem`(무기)에 `specialtyHand?: YahtzeeHand` 추가.
- 필살기 족보가 무기 특화 족보와 일치하면 `specialtyMultiplier` 적용.
- 무기 5종 → 각기 다른 특화 족보/스타일(예: 스트레이트 특화, 포카인드 특화 등). 데이터는 `constants.ts`에 정의.

## 7. 모바일 / 한 손 UX 원칙

- **세로 375×812 고정** 레이아웃 기준.
- **하단 엄지존(thumb-zone)** 에 핵심 조작(주사위 굴리기/족보 선택/필살기) 배치. 상단은 정보 표시.
- 터치 타겟 충분히 크게(최소 44px 권장).
- 한 손으로 전 과정(런 시작 → 전투 → 보상 선택 → 강화)이 가능해야 함.

## 8. 연출/피드백 (M4)

- 주사위 굴림 애니메이션, 홀드 하이라이트, 족보 성립 강조.
- 필살기 게이지 시각화(차오르는 바), 필살기 발동 연출, 데미지 팝업(`DamagePop` 재사용/강화).
- 사운드 / 햅틱(웹은 Vibration API, 안드로이드는 네이티브). **플랫폼 추상화**해서 끼워넣을 것(`05-CODING-GUIDELINES.md` 참고).

## 9. 비목표 (Non-goals)

- 백엔드/계정/실시간 멀티플레이/리더보드(서버) — 범위 밖.
- 외부 게임 자산 복사 — 금지.
- 과도한 수익화(가챠 BM 등)는 프로토타입 범위 밖(특성/보물 "획득"은 게임플레이 보상으로만).
